## Architectural decisions worth carrying forward

1. **The workbench preview is the canonical blade render** (per Ken's call during PR #82 review). Every other rendering surface should use it or be based on it. Custom hilt support, alternative quality tiers (lite/full/headless), and per-frame variations all branch from this single source of truth.

2. **`v2 glyph version byte is a SIGNAL, not a format change`** (Wave 7). The body schema didn't need to change because `diff/applyDelta` are generic. The version byte exists so old clients KNOW they may be missing data they should upgrade for, rather than silently stripping it.

3. **`apps/web/lib/blade/`** is the new home for shared blade-rendering code. `canvasHilt.ts` landed in PR #82. The deferred Phase 4 extraction should add `pipeline.ts` (capsule rasterizer) + `bloom.ts` (3-mip chain) + `glow.ts` (per-color profile) to it. Then `BladeCanvas` + `MiniSaber` + `FullscreenPreview` + `bladeRenderHeadless` + `card/drawBlade` all converge on these modules.

4. **Sidebar A/B layout is feature-flagged before flipped.** Phase 1 ships the wrapper behind `useABLayout: false` so Phase 2 (`blade-style` migration) can validate before the flip. Mobile fallback (Phase 3) is intentionally separate â€” needs a UX call on drawer vs bottom-sheet.

5. **Pre-launch posture is "validate-then-fix-as-v0.15.1-if-needed"** (per Ken's call during PR #71 prep). The hardware test gate was lifted from "blocks tag" to "post-tag confirmation." This unblocked the rest of tonight's work.

6. **Documentation-as-deliverable** for marketing copy. The `docs/launch/` directory has 7 polished copy-paste-ship files instead of vague guidelines. FILL-IN placeholders (Fett263 contact, specific YouTuber/TikTok names) are explicit, not implicit.
