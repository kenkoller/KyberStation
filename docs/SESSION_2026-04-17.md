## Key architectural decisions

### 1. Two `BladeConfig` types, drift-sentinel in tests
**Problem**: `codegen` package can't import from `engine` package at compile time because root `.npmrc` sets `node-linker=hoisted` + `symlink=false`. Workspace packages aren't linked into `node_modules`; combined with codegen's `rootDir: ./src`, importing engine types would require project references or a shared types package.

**Decision**: Codegen holds a mirror of engine's `BladeConfig`. A vitest typecheck assertion in `packages/codegen/tests/typeIdentity.test.ts` fails at compile time if the mirror drifts.

**Tradeoff**: Adding a new `BladeConfig` field means editing two files, but the drift check catches it immediately.

### 2. AST binding layer (`astBinding.ts`) as the unified façade
**Problem**: Config → AST → Code → AST → Config is the core WYSIWYG loop. Without a clean façade, features that tweak any step risked regressions in the others.

**Decision**: `packages/codegen/src/astBinding.ts` exports six "seams":
- `configToAST`, `astToCode`, `codeToAST`, `astToConfig` — the four direct transforms
- `syncFromConfig`, `syncFromCode` — pure binding-state transitions
- `hitToLED`, `positionToProffie`, `clamp01` — hit-test math for canvas clicks

### 3. Transition map as single source of truth (forward + inverse)
**Problem**: `ASTBuilder` had a switch for config → transition, `ConfigReconstructor` had a separate switch for transition → config, and they'd drifted. `standard` ignition emitted `TrWipeIn` but reconstructor mapped `TrWipe → standard` → blades silently round-tripped to the wrong type.

**Decision**: `packages/codegen/src/transitionMap.ts` holds `TRANSITION_MAPPINGS` with `{ id, kind, buildAST(ms), matches(node), extractMs(node) }` per entry. Forward and inverse share one data source. Covers all 19 ignition + 13 retraction IDs.

### 4. Lexer handles C++ `::` scope resolution
**Problem**: Tokenizer split `SaberBase::LOCKUP_NORMAL` into two tokens, which silently inflated `LockupTrL` from 4 to 5 args. The arg-count validator added in v0.10.0 exposed this.

**Decision**: Lexer's identifier loop now consumes `::` as part of an identifier.

### 5. GPL-3.0 attribution chain
- **KyberStation itself**: MIT (our own source)
- **Generated ProffieOS configs**: every emitted config.h starts with a header reminding the user that the firmware they compile is GPL-3.0
- **Community preset fixtures** (`packages/codegen/tests/fixtures/fett263/`): GPL-3.0 aggregate in the spirit of §5; remain MIT everywhere else in the repo
- **Full GPLv3 text**: `LICENSES/ProffieOS-GPL-3.0.txt`

### 6. Theme-token discipline
Goal: every colour in UI chrome goes through a CSS variable so themes and colour-blind modes work everywhere.

**CSS var namespaces** (all in `apps/web/app/globals.css`):
- `--accent`, `--accent-dim`, `--accent-border` — theme accent
- `--bg-deep`, `--bg-primary`, `--bg-secondary`, `--bg-surface`, `--bg-card` — backgrounds
- `--text-primary`, `--text-secondary`, `--text-muted` — text
- `--border-subtle`, `--border-light` — borders
- `--status-ok`, `--status-warn`, `--status-error`, `--status-info` — status (colour-blind overridable)
- `--faction-jedi`, `--faction-sith`, `--faction-grey`, `--faction-neutral` — preset factions (v0.9.1)
- `--era-prequel`, `--era-ot`, `--era-sequel`, `--era-animated`, `--era-eu` — preset eras
- `--badge-legends`, `--badge-creative`, `--badge-screen-accurate` — badge categories
- `--crystal-accent` — tracks the current blade's base colour (set by `useCrystalAccent`)

**Deliberately hardcoded**: `OLEDEditor.tsx` PIXEL_ON / PIXEL_OFF / GRID_COLOR simulate real monochrome OLED hardware and must not theme-track; a comment in the file explains.

### 7. GitHub-PR as the community gallery
**Decision** (confirmed this session): KyberStation does not run a backend. Community presets arrive as PRs to `packages/presets/src/characters/community/`. Moderation = PR review. See `docs/COMMUNITY_GALLERY.md` for the full plan.

### 8. WebUSB flash re-assessed as worth doing
**Original assessment** (this session morning): defer to Electron phase due to bricking risk.
**Revised assessment**: ProffieOS Workbench already proves WebUSB is safe for STM32 DFU on Proffieboard v3.9. A "use at your own risk" disclaimer + don't-unplug modal is industry-standard for the saber community. Target: v0.11.0, 4-6 hour focused sprint, `/flash` route, `webdfu` library.

## Lessons / patterns that worked

- **Parallel recon agents before execution**. Every large sprint started with 2-3 `Explore` agents mapping the problem, then serial execution. Kept sprints tight.
- **Plan mode for multi-phase work**. Complex sprints (v0.2.0, v0.9.1, v0.10.0) wrote detailed plan files first. Saved implementation thrash.
- **False-positive triage**. Agent audits would flag defects that spot-checking against source proved wrong. Recording them as "not fixing and here's why" saved future debate.
- **Verify-before-patch**. Several times a plan would cite specific line numbers; the actual file differed. Always open the file before editing.
