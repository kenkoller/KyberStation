# ProffieOS Flashing Guide

Comprehensive troubleshooting and reference for compiling, uploading, and
debugging ProffieOS configs generated by KyberStation / KyberStation. Every
section below was validated against real hardware (89sabers chassis,
Proffieboard V3.9, ProffieOS 7.x) during an end-to-end flashing session.

> **Shortcut (v0.11.0 and later):** KyberStation now ships an in-browser
> WebUSB flasher that handles steps 2–8 of the workflow below without
> Arduino CLI. Open the **Flash to Saber** panel in the Output tab, connect
> the Proffieboard in DFU mode, pick a firmware variant, and click Flash.
> See [`WEBUSB_FLASH.md`](./WEBUSB_FLASH.md) for the protocol-level
> reference and recovery procedure. The rest of this guide remains
> authoritative for the Arduino-CLI path and for everything downstream
> of the firmware flash (SD card layout, config.h debugging, voice pack,
> etc.).

## Pre-Flash Backup Checklist

Before flashing custom firmware to a Proffieboard for the first time, back up
everything. The STM32 has readout protection enabled -- you cannot extract
firmware or config data from the board after flashing.

- [ ] Copy the entire SD card contents to your Mac (all font folders, tracks,
  common, presets.ini).
- [ ] Save the original `config.h` file separately. If you did not receive one
  with the saber, ask the manufacturer or check their support resources. Once
  you flash, the original is gone.
- [ ] Save `presets.ini` if it exists on the SD card (Fett263 edit mode stores
  preset changes here).
- [ ] Note the original volume setting, LED count, prop file, and any Fett263
  defines.
- [ ] Photograph the SD card folder structure for reference.

## USB Connection

### Cable Selection

Many micro-USB cables are charge-only -- they carry power but have no data
lines. This is the single most common cause of "my board is not detected."

**Symptoms of a charge-only cable:**

- `system_profiler SPUSBDataType` returns nothing (zero USB devices listed).
- The chassis powers up (LEDs light, sounds play) but nothing appears in
  `/dev/cu.*`.
- `arduino-cli board list` shows no connected boards.

**Fix:** Swap the cable. If Cable #1 does not work, try Cable #2. Always keep
multiple known-good data cables on hand.

### Port and Connector Notes

- The Proffieboard's micro-USB port is on the PCB itself (the green circuit
  board). Some chassis (such as 89sabers) have a second port on the chassis
  body for charging. Use the one on the board, not the chassis port.
- Apple Silicon Macs (M1/M2/M3/M4) with USB-C ports need either a USB-C to
  micro-USB data cable or a USB-A adapter combined with a USB-A to micro-USB
  data cable.
- If one USB-C port on the Mac does not work, try a different one. Port behavior
  can vary.
- If the chassis powers up when you plug in USB (even without the kill switch
  engaged), that confirms the cable carries power. It does not confirm the cable
  carries data.

### Verifying a Successful Connection

A working data connection shows two things:

1. A serial device appears:

```bash
ls /dev/cu.usbmodem*
# Expected output: /dev/cu.usbmodemXXXX (e.g., /dev/cu.usbmodem14201)
```

2. `arduino-cli` detects the board:

```bash
arduino-cli board list
# Expected: a row showing "Proffieboard V3" or similar
```

If neither appears, swap cables and try again.

## Arduino CLI Setup and Commands

### Board Manager Installation

If you have not already installed the Proffieboard board manager:

```bash
arduino-cli config add board_manager.additional_urls \
  https://profezzorn.github.io/arduino-proffieboard/package_proffieboard_index.json

arduino-cli core update-index
arduino-cli core install proffieboard:stm32l4
```

### Fully Qualified Board Name (FQBN)

The FQBN for Proffieboard V3 is:

```
proffieboard:stm32l4:ProffieboardV3-L452RE
```

This FQBN alone is not sufficient. You must include board options or the
compile will fail.

### Required Board Options

| Option | Value         | Why                                                                |
|--------|--------------|--------------------------------------------------------------------|
| `dosfs` | `sdmmc1`    | Selects SDIO High Speed for SD card access. Without it, the build fails with `#error Tools->DOSFS should be set to SDIO`. |
| `usb`   | `cdc_msc`   | Enables USB mass storage alongside serial. Required if you want to send the `mass_storage` command over serial to access the SD card via USB. |

### Compile Command

```bash
arduino-cli compile \
  --fqbn "proffieboard:stm32l4:ProffieboardV3-L452RE:dosfs=sdmmc1,usb=cdc_msc" \
  "/path/to/ProffieOS"
```

The path should point to the ProffieOS directory containing the `.ino` file
and the `config/` directory with your `config.h`.

### Upload Command

```bash
arduino-cli upload \
  --fqbn "proffieboard:stm32l4:ProffieboardV3-L452RE:dosfs=sdmmc1,usb=cdc_msc" \
  --port /dev/cu.usbmodemXXXX \
  "/path/to/ProffieOS"
```

Replace `/dev/cu.usbmodemXXXX` with the actual serial port shown by
`arduino-cli board list`.

## Config.h Structure (ProffieOS 7.x)

ProffieOS includes `config.h` multiple times, each time with a different
preprocessor gate active. The compiler defines one gate, includes the file,
processes only the block under that gate, then repeats with the next gate.

The order of sections matters. Do not combine sections or move includes between
them.

```cpp
// ================================================================
// CONFIG_TOP -- Board includes, hardware defines, feature flags
// ================================================================
#ifdef CONFIG_TOP

#include "proffieboard_v3_config.h"

#define NUM_BLADES 1
#define NUM_BUTTONS 2
#define VOLUME 1200

// LED count -- REQUIRED. Without this, compilation fails with:
//   'maxLedsPerStrip' was not declared in this scope
#define maxLedsPerStrip 144

#define ENABLE_AUDIO
#define ENABLE_MOTION
#define ENABLE_WS2811
#define ENABLE_SD

// Fett263 prop defines (if using Fett263 prop file)
#define FETT263_EDIT_MODE_MENU
#define ENABLE_ALL_EDIT_OPTIONS    // Required when EDIT_MODE_MENU is defined
#define FETT263_MULTI_PHASE
#define FETT263_CLASH_STRENGTH_SOUND

#define CLASH_THRESHOLD_G 3.0

#endif  // CONFIG_TOP

// ================================================================
// CONFIG_PROP -- Prop file include (MUST be separate from CONFIG_PRESETS)
// ================================================================
#ifdef CONFIG_PROP

#include "../props/saber_fett263_buttons.h"

#endif  // CONFIG_PROP

// ================================================================
// CONFIG_PRESETS -- Preset definitions and blade hardware config
// ================================================================
#ifdef CONFIG_PRESETS

Preset presets[] = {
  { "font1", "tracks/track.wav",
    StylePtr<Layers<
      AudioFlicker<Blue, Rgb<0,0,128>>,
      BlastL<White>,
      SimpleClashL<White>,
      LockupTrL<Rgb<255,255,255>,
        TrConcat<TrInstant, White, TrFade<200>>,
        TrFade<400>,
        SaberBase::LOCKUP_NORMAL>,
      InOutTrL<TrWipe<300>, TrWipeIn<500>>
    >>(),
    "Preset1"
  },
};

BladeConfig blades[] = {
  { 0,
    WS281XBladePtr<144, bladePin, Color8::GRB,
      PowerPINS<bladePowerPin2, bladePowerPin3>>(),
    CONFIGARRAY(presets)
  }
};

#endif  // CONFIG_PRESETS

// ================================================================
// CONFIG_BUTTONS -- Physical button pin assignments
// ================================================================
#ifdef CONFIG_BUTTONS

Button PowerButton(BUTTON_POWER, powerButtonPin, "pow");
Button AuxButton(BUTTON_AUX, auxPin, "aux");

#endif  // CONFIG_BUTTONS
```

### Critical Ordering Rules

**The prop `#include` MUST be in `CONFIG_PROP`, not in `CONFIG_PRESETS`.** If
you place the prop include inside CONFIG_PRESETS, the prop's header file
(`prop_base.h`) will reference the `blades[]` array before it has been
defined, producing a compilation error. The CONFIG_PROP section is processed
in a separate pass, which is why it exists as a distinct gate.

**`maxLedsPerStrip` MUST be defined in `CONFIG_TOP`.** If omitted, every file
that references LED strip length will fail with `'maxLedsPerStrip' was not
declared in this scope`.

**`ENABLE_ALL_EDIT_OPTIONS` MUST be defined when `FETT263_EDIT_MODE_MENU` is
defined.** Without it, the edit mode menu will not compile.

## Code Generation Bugs and Fixes

These are actual compilation errors encountered when flashing KyberStation-
generated configs. Each entry describes the error, the cause, and the
correction that the codegen module must apply.

### 1. Missing `maxLedsPerStrip`

**Error:**
```
error: 'maxLedsPerStrip' was not declared in this scope
```

**Cause:** CONFIG_TOP did not include `#define maxLedsPerStrip <count>`.

**Fix:** Always emit `#define maxLedsPerStrip <count>` in CONFIG_TOP, where
`<count>` matches the LED count in the WS281XBladePtr template argument.
The KyberStation ConfigBuilder now emits this as
`const unsigned int maxLedsPerStrip = 144;`.

### 2. Missing `SaberBase::` Prefix on Lockup Enums

**Error:**
```
error: 'LOCKUP_NORMAL' was not declared in this scope
error: 'LOCKUP_DRAG' was not declared in this scope
```

**Cause:** Lockup type enums were emitted without their namespace prefix.

**Fix:** Always use fully qualified names:
- `SaberBase::LOCKUP_NORMAL`
- `SaberBase::LOCKUP_DRAG`
- `SaberBase::LOCKUP_LIGHTNING_BLOCK`
- `SaberBase::LOCKUP_MELT`

### 3. Wrong Template Name: `TrWipeSparkTipL`

**Error:**
```
error: 'TrWipeSparkTipL' was not declared in this scope
```

**Cause:** The template is named `TrWipeSparkTipX`, not `TrWipeSparkTipL`.
The `X` suffix denotes that it takes a template parameter for the spark color.

**Fix:** Use `TrWipeSparkTipX` in all codegen output.

### 4. Missing `Int<>` Wrapper on Integer Template Arguments

**Error:**
```
error: type/value mismatch at argument 2 of template 'TrWipeSparkTipX'
```

**Cause:** Raw integer literals like `500` were passed where ProffieOS expects
a type. Many ProffieOS templates take "function" types, not raw values.

**Wrong:**
```cpp
TrWipeSparkTipX<White, 500>
```

**Correct:**
```cpp
TrWipeSparkTipX<White, Int<500>>
```

**Fix:** Wrap integer arguments in `Int<>` when the template parameter
expects a function type. This applies to transition duration arguments in
templates with the `X` suffix.

### 5. Prop Include in Wrong Section

**Error:**
```
error: 'blades' was not declared in this scope
(from prop_base.h, during CONFIG_PRESETS processing)
```

**Cause:** The `#include "../props/saber_fett263_buttons.h"` was placed inside
`#ifdef CONFIG_PRESETS` instead of `#ifdef CONFIG_PROP`.

**Fix:** Always emit prop includes in a dedicated `#ifdef CONFIG_PROP` block,
separate from CONFIG_PRESETS.

### 6. Missing `ENABLE_ALL_EDIT_OPTIONS`

**Error:**
```
error: ... (edit mode menu references fail)
```

**Cause:** `FETT263_EDIT_MODE_MENU` was defined but `ENABLE_ALL_EDIT_OPTIONS`
was not.

**Fix:** When emitting `FETT263_EDIT_MODE_MENU`, always also emit
`#define ENABLE_ALL_EDIT_OPTIONS` before it.

## LED Count

### How to Determine LED Count

The LED count must match the physical number of LEDs in the blade. If the
count is too low, the top portion of the blade will remain dark.

**Symptom:** The top ~1/6 of the blade is dark. If you set 120 LEDs and the
blade actually has 144, approximately 24 LEDs at the tip will not light.

**Standard neopixel blade LED counts by length:**

| Blade Length | Typical LED Count |
|-------------|-------------------|
| 24"         | 88                |
| 28"         | 103               |
| 32"         | 117               |
| 36"         | 132 -- 144        |
| 40"         | 147               |

The 89sabers chassis with a 36" blade uses 144 LEDs.

### Where LED Count Appears in Config

LED count must be set in two places and they must match:

1. In CONFIG_TOP:
```cpp
#define maxLedsPerStrip 144
```

2. In the BladeConfig inside CONFIG_PRESETS:
```cpp
WS281XBladePtr<144, bladePin, Color8::GRB, PowerPINS<bladePowerPin2, bladePowerPin3>>()
```

If these values differ, behavior is undefined. Always keep them in sync.

## Volume

The `VOLUME` define in CONFIG_TOP controls maximum volume (0--3000 range).

| Value | Result                                                     |
|-------|------------------------------------------------------------|
| 2000  | Default in many configs. Clips and distorts on 89sabers chassis speakers. |
| 1200  | Good starting point for testing. Clear without distortion.  |
| 800   | Quiet. Good for late-night testing.                         |

```cpp
#define VOLUME 1200
```

Note: The 89sabers factory firmware shipped with `VOLUME 2000`. The distortion
is present in the factory config as well -- it is not caused by custom firmware.

## Sound Fonts and SD Card Requirements

### Minimum Font Folder Contents

ProffieOS loads sounds from named WAV files in each font's directory. While
ProffieOS supports a `common/` folder as a fallback for shared sounds, this
did not work reliably in testing. Place all required sounds directly in each
font folder.

**Essential files per font folder:**

| File(s)                         | Purpose                              |
|---------------------------------|--------------------------------------|
| `hum01.wav`                     | Continuous hum while blade is on     |
| `in01.wav`, `in02.wav`, ...     | Ignition sounds                      |
| `out01.wav`, `out02.wav`, ...   | Retraction sounds                    |
| `clsh01.wav` -- `clsh16.wav`    | Clash sounds (randomized)            |
| `swng01.wav` -- `swng16.wav`    | Swing sounds (legacy mode)           |
| `swingh01.wav` -- `swingh08.wav`| SmoothSwing high pair                |
| `swingl01.wav` -- `swingl08.wav`| SmoothSwing low pair                 |
| `blst01.wav` -- `blst08.wav`    | Blast deflection sounds              |
| `lock01.wav`                    | Lockup sound                         |
| `drag.wav`                      | Drag effect sound                    |
| `stab01.wav`                    | Stab effect sound                    |
| `smoothsw.ini`                  | SmoothSwing configuration            |

### Preset Identification Sounds

Two boot-related sound files are commonly confused:

- **`boot01.wav`** -- Plays once when the board first powers up (battery
  connected or kill switch engaged). Only plays for the first preset.
- **`font.wav`** -- Plays when switching TO this preset. This is the file
  used for preset identification, not `boot01.wav`.

### Track Field in Preset Definitions

The second field in each preset entry is the background music track path:

```cpp
{ "fontdir", "tracks/track.wav", StylePtr<...>(), "PresetName" }
//           ^^^^^^^^^^^^^^^^^^ track path
```

**If no track file exists, use an empty string:**

```cpp
{ "fontdir", "", StylePtr<...>(), "PresetName" }
```

Do not point this to a non-existent file. If the path references a WAV file
that is not on the SD card, ProffieOS will play an error sound every time
that preset loads.

## Fett263 Voice Pack

### The Problem

The Fett263 prop file (`saber_fett263_buttons.h`) initializes a `SoundLibrary`
object that calls `require_version(1)` at startup. On every preset change, the
prop checks for `SFX_mnum` (numbered voice menu files) and `voicepack.ini`.

If these files are not found, ProffieOS plays a "Voice pack not found" error
sound on every preset switch. This happens even without `EDIT_MODE_MENU`
enabled -- the SoundLibrary initialization runs regardless.

### Solutions (in order of preference)

#### Option 1: Disable the Check in Source (Cleanest for Testing)

Edit `ProffieOS/sound/sound_library.h` and replace the body of the
`CheckVersion()` method with a comment or empty block. This eliminates the
voice pack dependency entirely. Suitable for development and testing configs
where edit mode functionality is not needed.

#### Option 2: Install the Voice Pack (Required for Production)

1. Download the English Voice Pack from:
   https://fredrik.hubbe.net/lightsaber/sound/ (Voice Pack A through F)

2. Extract the `common/` folder to the SD card root.

3. **Create `common/voicepack.ini` manually.** The download does not include
   this file, and ProffieOS requires it. The file must contain exactly:

```ini
voice_pack_version=1
```

4. Verify the folder structure matches the layout below.

#### Option 3: Use a Different Prop File

`saber_sa22c_buttons.h` does not require voice packs. However, it changes
the button mapping: retraction requires a medium hold of the power button
instead of a short click. This alters the user experience significantly.

### Voice Pack Folder Structure

```
SD Card Root/
  common/
    voicepack.ini              <-- MUST contain: voice_pack_version=1
    mnum/
      mnum1.wav
      mnum2.wav
      ...
      mnum20.wav
    clrlst/
      clrlst01.wav
      clrlst02.wav
      ...
      clrlst27.wav
    maccept.wav
    mcancel.wav
    ... (100+ additional menu prompt files)
```

## USB Mass Storage Mode

USB mass storage allows the Mac to mount the Proffieboard's SD card as a
removable volume over USB, without physically removing the card.

### Requirements

- The firmware must be compiled with `usb=cdc_msc` in the FQBN board options.
  If the firmware was compiled without this option, mass storage commands will
  not work. Recompile and re-flash with the option enabled.

### Entering Mass Storage Mode

Send the `mass_storage` command over the serial port:

```bash
printf "mass_storage\r\n" > /dev/cu.usbmodemXXXX
```

The SD card will mount as a removable volume on the Mac.

### Limitations

- **Write speed is slow.** Approximately 4 folders per minute when each folder
  contains around 80 files. For bulk operations (copying entire font libraries),
  physically removing the SD card and using the Mac's built-in SD card reader
  is roughly 10x faster.
- **The board cannot play sounds while in mass storage mode.** The Mac owns the
  filesystem; the board's firmware cannot access it simultaneously.
- **To exit mass storage mode,** unplug the USB cable. The board returns to
  normal operation on the next power cycle.

## Quick Reference: Complete Flash Workflow

For reference, here is the end-to-end sequence for a clean flash:

```
1. Back up SD card and original config.h
2. Connect Proffieboard via USB data cable
3. Verify connection:
     ls /dev/cu.usbmodem*
     arduino-cli board list
4. Place config.h in ProffieOS/config/
5. Compile:
     arduino-cli compile \
       --fqbn "proffieboard:stm32l4:ProffieboardV3-L452RE:dosfs=sdmmc1,usb=cdc_msc" \
       "/path/to/ProffieOS"
6. Fix any compilation errors (see Code Generation Bugs section)
7. Upload:
     arduino-cli upload \
       --fqbn "proffieboard:stm32l4:ProffieboardV3-L452RE:dosfs=sdmmc1,usb=cdc_msc" \
       --port /dev/cu.usbmodemXXXX \
       "/path/to/ProffieOS"
8. Test: ignite blade, switch presets, trigger effects
9. If voice pack errors on preset switch, see Fett263 Voice Pack section
```

## Troubleshooting Checklist

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Mac shows zero USB devices | Charge-only cable | Swap to a known data cable |
| Board powers up but no serial device | Charge-only cable, or wrong USB port on chassis | Use the port on the PCB, try different cable |
| `#error Tools->DOSFS should be set to SDIO` | Missing `dosfs=sdmmc1` in FQBN | Add `dosfs=sdmmc1` to board options |
| `'maxLedsPerStrip' was not declared` | Missing define in CONFIG_TOP | Add `#define maxLedsPerStrip 144` |
| `'LOCKUP_NORMAL' was not declared` | Missing `SaberBase::` prefix | Use `SaberBase::LOCKUP_NORMAL` etc. |
| `'TrWipeSparkTipL' was not declared` | Wrong template name | Use `TrWipeSparkTipX` |
| `type/value mismatch at argument N` | Raw integer where type expected | Wrap in `Int<>`: `Int<500>` |
| `'blades' was not declared` (from prop_base.h) | Prop include in wrong section | Move `#include` to `#ifdef CONFIG_PROP` |
| Top of blade is dark | LED count too low | Increase `maxLedsPerStrip` and WS281XBladePtr count |
| Audio clips and distorts | Volume too high | Lower `#define VOLUME` to 1200 |
| "Voice pack not found" on preset change | Missing Fett263 voice pack | Install voice pack or disable check (see section) |
| Error sound on preset load | Track path points to missing file | Use `""` for track if no track WAV exists |
| `mass_storage` command does nothing | Firmware compiled without `usb=cdc_msc` | Recompile with `usb=cdc_msc` in FQBN |
