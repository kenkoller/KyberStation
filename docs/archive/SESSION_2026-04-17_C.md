### v0.11.1 scope breakdown (PR #19)

**WS2 — Alert-color discipline.** Reserved `#ff4444` / `rgba(255,60,60,…)` for actual error semantics. Tokenised:

- `.era-*` classes → `rgb(var(--era-*))` so colorblind + Mustafar theme overrides flow through
- `@keyframes retract-breathe` → `rgb(var(--badge-creative))` amber (retract is an action warning, not an error)
- `.console-alert` → `rgb(var(--status-error))` (still red, but tokenised)
- `.sw-sith-text` / `.sw-jedi-text` gradient stops → new `--faction-sith-deep` / `--faction-jedi-deep` tokens
- HUD raw hex in `PowerDashboard.tsx` + `EngineStats.tsx` (`#22c55e` / `#eab308` / `#ef4444`) → `rgb(var(--status-ok/warn/error))`
- R/G/B-channel viz in `BladeCanvas`, `VisualizationStack`, `RGBGraphPanel`, `visualizationTypes.ts` **intentionally preserved** — those are literal channel renders, not alert semantics

**WS3 — Skeleton + ErrorState coverage.** New `<ErrorState>` primitive with 4 named variants:

```tsx
<ErrorState
  variant="load-failed" | "parse-failed" | "save-failed" | "import-failed"
  message={string}
  onRetry={() => void}
  compact={boolean}
/>
```

Applied across panels with async boundaries:

- In-panel `<ErrorState>` + retry: CommunityGallery, PresetBrowser, CodeOutput (`.kyberstation.json` import), OLEDEditor (BMP import)
- Toast-based transient failure: SaberProfileManager, SoundFontPanel, PresetGallery
- CardWriter + FlashPanel intentionally unchanged — their existing state machines are fit-for-purpose

**WS4 — Color-glyph pairing.** New `<StatusSignal>` + `<EraBadge>` + `<FactionBadge>` in `apps/web/components/shared/StatusSignal.tsx`. Pairs color-coded indicators with typographic glyphs for deuteranopia/protanopia/tritanopia redundancy:

- Status variants: idle `●` / active `◉` / success `✓` / warning `▲` / alert `⚠` / error `✕`
- Era monograms: prequel `◇` / OT `◆` / sequel `▲` / animated `◯` / EU `✦` / legends `✧`
- Faction markers: Jedi `☉` / Sith `✦` / Grey `◐` / Neutral `·`

Applied in:

- `StatusBar` — power/storage/LED indicators pair glyph with color
- `PresetGallery` — `<FactionBadge>` on preset entries
- `EngineStats` — FPS readout prefixed with `✓`/`▲`/`✕` based on performance bucket
- `PowerDashboard` — system-status header gains `<StatusSignal>` next to animated `ConsoleIndicator`

`TimelinePanel` event-type colors (ignite/retract/clash/etc.) stayed as raw hex by design — those are distinct identity colors already paired with text labels, not alert semantics.

### PR #22 scope (branch protection)

Client-side defense against the `git push -f origin main` footgun, until KyberStation upgrades to GitHub Pro (server-side branch protection is paywalled on private repos):

- `.githooks/pre-push` — blocks force-push + branch-deletion pushes to `refs/heads/main`. Other branches untouched. Bypass via `git push --no-verify` for rare intentional cases.
- `scripts/install-git-hooks.mjs` — idempotent `git config core.hooksPath .githooks` setup. Every contributor runs `pnpm run hooks:install` once per clone.
- `scripts/setup-branch-protection.mjs` — post-upgrade automation that creates a `main-protection` ruleset via `gh api` when GitHub Pro unlocks the rulesets endpoint.
- `docs/BRANCH_PROTECTION.md` — explains why we need client-side hooks (paywall), how to install, how to bypass, and migration paths to real server-side protection.

**Merge-order playbook that worked:**

1. Check `gh pr view <n> --json mergeable,mergeStateStatus,statusCheckRollup` before squash-merging anything
2. Diff each PR's file footprint against `main` + against the other open PRs before ordering
3. Merge smallest / most isolated PR first; let bigger ones rebase onto clean bases
4. Never merge a PR whose authoring session has uncommitted work in its worktree — that ships a half-finished feature

**Phase 1 merge order used:** PR #22 (branch protection, 4 files, isolated) → PR #21 (naming math, 3 files, isolated). PR #20 (Kyber Crystal) intentionally held pending Session A's expansion. PR #23 (naming polish) still CLEAN when this session closed.

## The generic wrap-up prompt

Any session can use this to close gracefully. Originally drafted in-session; formalising here so future sessions inherit the pattern:

```
Wrap-up pass. Please:

1. Status check: run `git status` and `git log --oneline origin/main..HEAD`.
   Report (a) uncommitted files, (b) unpushed commits, (c) what branch
   you're on, (d) whether a PR exists for it.

2. Separate ship-ready from WIP. For every uncommitted file, decide:
   - SHIP: complete, tested, coherent with the PR's scope → stage + commit
   - DEFER: incomplete, experimental, or out of scope → add to the
     deferred-work list (step 4) and either `git stash` it or leave it
     uncommitted with a note explaining why

3. Land the ship-ready work:
   - Run `pnpm -w typecheck && pnpm -w test` — must pass before committing
   - Commit with a conventional-commits message describing what landed
   - Push to your remote branch
   - If no PR exists, open one with `gh pr create` targeting `main`.
     If a PR exists, add a comment summarizing this batch of commits.

4. Document deferred work explicitly. For each DEFER item from step 2,
   add an entry in the PR description (or a new `### Deferred` section
   in CHANGELOG.md under [Unreleased]) with:
   - What it is (one line)
   - Why it's deferred (blocked, out of scope, needs more design, etc.)
   - Who should pick it up (this session later / dedicated sprint / anyone)

5. Reply with a short report: (a) merge-readiness verdict (READY /
   NEEDS-MORE-WORK / BLOCKED), (b) PR URL, (c) the deferred list, (d)
   any conflicts you hit with `origin/main`.

Do NOT merge your own PR. Wait for the orchestrator to decide merge order.
```
